# 重构任务模板

**模板类型**: 任务流程
**流程步骤**: 现状分析 → 架构设计与迁移计划 → 分阶段实施 → 测试验证 → 文档更新

---

## 第一步：现状分析

你是一个专业的系统架构师，需要分析当前代码库的现状，为架构重构做准备。

### 分析要点

1. **代码结构分析**
   - 分析当前代码的组织方式，识别需要重构的模块
   - 识别需要保留的模块

2. **依赖关系分析**
   - 分析模块间的依赖关系，绘制依赖图
   - 识别循环依赖和紧耦合

3. **耦合点识别**
   - 找出代码中的耦合点
   - 识别违反单一职责原则、依赖倒置原则的地方

4. **受影响文件清单**
   - 列出所有需要迁移、修改或删除的文件，按优先级分类

5. **潜在风险识别**
   - 识别重构过程中可能遇到的技术风险、业务风险、兼容性风险

## 第二步：架构设计与迁移计划

你是一个专业的系统架构师，需要基于现状分析同时设计**目标架构**并给出**可执行的迁移计划**。

### 设计要点

1. **目标架构设计**
   - 新的目录结构设计
   - 模块划分方案（按职责、按层次、按功能等）
   - 依赖关系设计（消除循环依赖，降低耦合度）
   - 接口抽象设计（定义清晰的接口边界）

2. **迁移总体策略**
   - 分阶段迁移策略（整体思路与关键原则）
   - 迁移顺序（识别关键路径，确定迁移优先级）
   - 回滚方案（如果出现问题如何快速回滚）

3. **阶段划分**
   - 每个阶段的名称和目标
   - 每个阶段的具体任务清单与依赖关系
   - 每个阶段的验收标准

4. **文件迁移清单**
   - 关键文件/目录的源路径和目标路径
   - 文件迁移时需要修改的内容的典型示例
   - 文件或模块的迁移优先级

5. **风险评估与应对**
   - 技术风险评估
   - 业务风险评估
   - 应对措施

---

## 第三步：分阶段实施

你是一个专业的软件开发工程师，需要基于「架构设计与迁移计划」进行分阶段实施。

### 实施要点

1. **当前阶段实施**
   - 根据迁移计划，执行当前阶段的任务
   - 按照文件迁移清单，逐个迁移或重构文件
   - 更新导入路径、引用关系和依赖关系

2. **代码实现**
   - 创建新的目录结构（如需要）
   - 迁移代码到新位置（如需要）
   - 重构代码以符合新架构
   - 更新所有导入路径、引用和依赖

3. **问题处理**
   - 记录实施过程中遇到的问题
   - 提供解决方案和替代方案

4. **进度跟踪**
   - 记录已完成的文件和任务
   - 更新实施进度和完成度

---

## 第四步：测试验证

你是一个专业的测试工程师，需要基于重构后的代码进行全面的测试验证。

### 测试要点

1. **构建验证**
   - 验证新架构下的构建脚本是否正常工作
   - 验证所有包能否正确构建
   - 验证依赖关系是否正确

2. **功能测试**
   - 设计功能测试用例
   - 执行功能测试
   - 验证所有功能是否正常

3. **集成测试**
   - 验证模块间的集成是否正常
   - 验证接口调用和消息通信是否正常

4. **回归测试**
   - 执行完整的回归测试套件
   - 验证原有功能未受影响

5. **性能测试**
   - 测试启动时间、响应时间、内存使用
   - 对比重构前后的性能指标

---

## 第五步：文档更新

你是一个专业的文档管理专家，需要基于重构结果更新相关文档。

### 更新要点

1. **架构文档更新**
   - 更新架构设计文档，反映新的架构结构
   - 更新目录结构说明
   - 更新模块划分说明
   - 更新依赖关系图

2. **API文档更新**
   - 更新公共API文档
   - 更新接口定义文档
   - 更新废弃API的迁移指南

3. **迁移指南**
   - 编写详细的迁移指南
   - 说明如何从旧架构迁移到新架构
   - 提供常见问题解答

4. **变更日志**
   - 记录所有文件迁移
   - 记录所有配置变更
   - 记录破坏性变更
